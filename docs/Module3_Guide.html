Spatial Data Integration & Overlay Analysis

Module Overview and Objectives

So far, your hardship index is built entirely from census variables — measures of who lives in a place. But neighborhood quality is also shaped by what is near a place. Proximity to environmental hazards, grocery stores, transit stops, or health clinics matters as much as demographic composition. This module teaches you to combine tabular census data with external point data using spatial operations.

The central challenge is that these data sources live in different formats, different coordinate reference systems, and different geographic units. You will learn to align them using spatial joins, buffer analysis, and coordinate transformations — the core operations of spatial data integration. Along the way, you will confront the Modifiable Areal Unit Problem (MAUP), one of the most important methodological issues in spatial analysis.

After completing this module, you will be able to:

  • Explain the role of coordinate reference systems (CRS) and transform data between projections using st_transform()
  • Perform point-in-polygon spatial joins using st_join() to count point features within census boundaries
  • Conduct buffer analysis using st_buffer() to measure proximity to spatial features
  • Use st_intersection() to clip and combine spatial layers
  • Describe the Modifiable Areal Unit Problem and its implications for spatial analysis
  • Create an enriched hardship index that incorporates spatial access variables alongside census demographics
  • Integrate external data sources (e.g., EPA facility data, USDA food access data) with census geographies

Section 1: Coordinate Reference Systems

Before you can combine spatial datasets, they must share the same coordinate reference system (CRS). A CRS defines how locations on the curved Earth are represented on a flat surface. Two data layers that appear to be in the same location but use different CRS definitions will not align — your spatial joins will produce nonsense results.

Geographic vs. Projected CRS

  • Geographic CRS (e.g., WGS 84 / EPSG:4326): Uses latitude and longitude in degrees. Good for global data and web mapping. Distances and areas are distorted because degrees are not constant-length units
  • Projected CRS (e.g., NAD83 UTM Zone 12N / EPSG:26912): Projects the globe onto a flat surface. Distances and areas are accurate within the projection's valid zone. Essential for buffer analysis and area calculations

Practical Rule

Always check CRS with st_crs() before performing any spatial operation. If your layers have different CRS values, use st_transform() to reproject one to match the other. For operations that require accurate distances (buffers, area calculations), use a projected CRS appropriate for your study area.

Common mistake: Creating a buffer of "1000" on data in EPSG:4326. This produces a buffer of 1000 degrees, not 1000 meters. Always project first.

Section 2: Spatial Joins and Overlay Operations

A spatial join attaches attributes from one spatial layer to another based on their geographic relationship — analogous to a table join in SQL, but using location instead of a key column.

Point-in-Polygon Joins

The most common spatial join in community analytics: "How many [facilities/events/resources] fall within each [census tract/block group]?" The st_join() function in the sf package performs this operation:

joined <- st_join(polygons, points, join = st_intersects)

After joining, you typically group by the polygon ID and count the number of matched points.

Buffer Analysis

Buffers create a zone of specified distance around a feature. Common uses include:

  • Identifying census tracts within 1 mile of a hazardous facility
  • Counting the number of grocery stores within 0.5 miles of each neighborhood centroid
  • Measuring overlap between service areas and target populations

In R: st_buffer(points, dist = 1609) creates a 1-mile (1609 meter) buffer around each point (assuming a projected CRS in meters).

Intersection and Clipping

st_intersection() returns only the portions of features that overlap between two layers. This is useful for clipping a statewide dataset to your study area or computing the area of overlap between buffers and census geographies.

Section 3: The Modifiable Areal Unit Problem (MAUP)

The MAUP is a fundamental issue in spatial analysis: the results of statistical analyses can change depending on how geographic areas are defined. It has two components:

  • Scale effect: Aggregating data to larger units (e.g., from block groups to tracts to counties) changes summary statistics. Correlations that appear at one scale may vanish at another
  • Zoning effect: Even at the same scale, different boundary configurations can produce different results. Congressional districts drawn two different ways yield different demographic profiles

Why it matters for this course: When you count environmental facilities per census block group, the result depends entirely on where the Census Bureau drew the block group boundaries. A facility sitting right on a boundary could be counted in either unit. The hardship index values themselves are sensitive to the choice of geographic unit.

There is no perfect solution to the MAUP, but awareness is the first defense. Best practices include: analyzing at multiple scales, using the finest available geography, and being transparent about the units used.
